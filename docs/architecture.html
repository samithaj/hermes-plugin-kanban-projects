01The idea in one breath

You run many Claude Code agents at once, in different repos. Some finish, some block on a permission prompt, some quietly idle with no next step. The hard part isn't running them — it's noticing which one needs you and getting back into it fast. This system closes that gap two ways:

🔊 The audio half

Per-session Claude Code hooks (in ~/.hermes/hooks, not this repo) play a repeating sound the moment an agent hits a permission/idle prompt, and stop it when you respond. Your ears are the interrupt.

👁️ The visual half (this plugin)

A Kanban dashboard grouped by project that ranks every needs‑you session into a triage strip, each with an Open in Warp / Send button. Your eyes pick the target; one click navigates.

Note: AskUserQuestion fires no hook, so the sound is silent for it — the visual triage strip is the backstop for that gap.

Two data planes meet in the dashboard. Down the middle: agents → a mirror board → the dashboard (the "what's happening" plane). Back up the right: your click → the Warp bridge → Warp → the resumed agent (the "act on it" plane). The big arrow is the loop closing.

03The moving parts

Agent sessions

Ordinary Claude Code runs, one per repo/task. Each writes a job state file (~/.claude/jobs/<id>/state.json) and a transcript (~/.claude/projects/**/<uuid>.jsonl). These are the source of truth the rest reads.

sync_claude_kanban.py

A cron job (~60 s, in ~/.hermes/scripts) that reads jobs + transcripts and writes the claude-code-work mirror board. It packs each session's signals onto one body line: State · Process · Waiting · Needs + task progress. This plugin never writes that board.

Bundled Kanban API

Same-origin REST at :9120/api/plugins/kanban/* over a small DB. The plugin rides this for all persistence — custom plugins can't ship a backend, so there is no server code here.

kanban-projects plugin

Hand-authored plain JS (dashboard/dist/index.js, hyperscript h(), no build step). Reads both boards via SDK.fetchJSON, joins them by full cwd path, and renders the triage UI.

Warp bridge (:9777)

A tiny local HTTP server (warp_kanban_bridge.py, symlinked into ~/.hermes/scripts). Turns a dashboard click into a real Warp tab running the right claude command. Details below.

CUA-driver & Warp

The bridge can't type into a terminal via osascript (TCC blocks the detached daemon), so it uses cua-driver (Accessibility) to type into Warp, or opens a .command launcher via warp://action/new_tab.

04Data model — two boards, one join key

BoardOwnerWhat it holds
claude-code-workmachine (sync script) One card per live/finished agent session — its state, process, waiting flag, task-list progress.
project-goalshuman (this plugin) One card per project — objective, next step, "done when", triage bucket, Obsidian note path.

They're joined by a canonical project key = the full cwd path (not the basename — basenames collide). The goal card body is markdown so it stays editable in the stock Kanban too:

Path: /Users/sam/Documents/dev/<project>      ← the join key, must stay first
Bucket: now|next|later|blocked|done          ← human triage lives HERE, not in status
Obsidian: [[wiki/projects/<slug>]]
Note path: wiki/projects/<slug>.md
**Objective:** …
**Next:** …
**Done when:**
- [ ] …
🔒 Why the bucket lives in the body, never the status. The Hermes dispatcher scans every non-archived board each tick and auto-promotes todo→ready; a ready+assigned card gets executed by a worker. So every project-goals card must stay status: triage and unassigned, or a "goal" would silently run as a job.

05The Warp bridge — how a click becomes a session

The triage strip's buttons are just links to the bridge. Three endpoints:

EndpointButtonEffect
GET /open/<id>Open in WarpFocus the session's existing Warp tab, or open a fresh one.
GET /compose/<id>?text=DraftOpen + type the text as a draft — Enter withheld, you review & send.
GET /send/<id>?text=Send to ClaudeOpen + deliver the prompt and submit it.

Choosing the right command

An <id> can be a short Agent-View job id or a full session UUID. The bridge resolves it and picks:

Session shapeCommand
Live Agent-View jobclaude attach <short-id>
Terminal (done/stopped/failed) with a real transcript, or a full interactive UUIDclaude --resume <uuid> [<prompt>]
Failed with no resumable transcriptfresh claude, seeded with the saved summary
Prompt delivery without leaks. A cold send passes the prompt as claude --resume <uuid> "$(cat <file>; rm -f <file>)" — the text rides a short-lived file that the shell reads and deletes inline, so it never lands in shell history and self-cleans. The command runs from a .command launcher (opened via warp://action/new_tab) so keystrokes can't leak into whatever tab is currently frontmost.

Each launched tab records a focus URL (warp://session/<uuid>) so a later click re-focuses the same tab instead of spawning duplicates.

06"Send to Claude", step by step

  1. You type a prompt in the triage panel and click Send → the browser hits :9777/send/<job>?text=….
  2. The bridge resolves the session and refuses only if it's finished with nothing resumable; otherwise it decides attach vs resume vs fresh.
  3. For a resumable/full-UUID send it writes the prompt to a temp file, builds claude --resume <uuid> "$(cat f; rm -f f)", and writes a .command launcher.
  4. It opens the launcher with warp://action/new_tab?path=…. The launcher cds into the session's cwd, records the focus URL, then execs the resume command.
  5. The bridge waits for the focus record (proof the tab actually came up) before returning {submitted:true} — a failed launch reports 504, never a false success.
  6. Claude resumes in the right directory, receives your prompt as its first message, and continues. The loop is closed.

07Session persistence (so resume even works)

Hermes launches these agents as children, so they inherit CLAUDE_CODE_CHILD_SESSION=1, which normally excludes them from agent-view/background persistence — meaning they couldn't be backgrounded or re-attached. The documented override re-enables it:

export CLAUDE_CODE_FORCE_SESSION_PERSISTENCE=1     # in ~/.zshenv

It lives in ~/.zshenv (not .zshrc) because every bridge launch path is a zsh login shell — including the non-interactive zsh -lc — and only .zshenv is read by all of them. Effect: new Hermes sessions persist and show up attachable in Agents view.

08Invariants (don't break these)

UI-only

No backend here. All persistence rides the bundled Kanban API via SDK.fetchJSON. Never add plugin_api.py.

Goals never dispatchable

Create goal cards with {triage:true}; edit with {body}; never set status or an assignee.

Ship dist as-is

Edit dashboard/dist/index.js directly in hyperscript; keep node --check green. No build step.

Don't write the mirror

claude-code-work is owned by the sync script. Read it; never write it from here.

Obsidian is link-only

Render obsidian:// links from goal metadata. The plugin never writes the vault.

Bounded signal parsing

Parse session fields with a bounded regex; a greedy line capture once swallowed the blocked/waiting flags.

09Recent hardening

An xhigh code review of the resume/send path surfaced nine defects; the fixes shipped on main:

  • Resume-with-prompt delivery now works for full session UUIDs and terminal-resumable jobs (the prompt rides the self-cleaning temp file).
  • Drafts no longer auto-submit — baking the prompt into the command is gated on submit.
  • No false "sent" — the send waits for confirmed focus before reporting success.
  • Leaked prompt files swept; repeated transcript scans are cached per request.
  • Removed the expect resume-menu wrapper — running Claude's TUI through an expect pty corrupted the terminal handshake (stray ^[P>|Warp… / ?62c). Resuming directly is clean; the resume-summary menu is suppressed at the Claude-Code level ("Don't ask again").
